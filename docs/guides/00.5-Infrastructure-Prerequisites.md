# TrueNAS Pool Architecture

TrueNAS at `10.0.10.80` uses three ZFS pools, each with a distinct purpose:

| Pool | Purpose |
|---|---|
| `tera` | Bulk cold storage — large media files, archives, rarely-accessed data |
| `core` | Hot storage — Kubernetes persistent volumes, actively read/written by the cluster |
| `archive` | Backup storage — etcd snapshots, Velero data via MinIO, host backups |

All datasets on these pools are mounted at `/mnt/<pool>/<dataset>` on TrueNAS, which is the path
used when configuring NFS exports and host paths.

```
TrueNAS ZFS Layout
├── core/
│   └── k8s_volumes/          ← Kubernetes PVCs (provisioned dynamically by NFS provisioner)
├── archive/
│   └── backups/              ← NFS-exported backup parent
│       ├── k8s/              ← Kubernetes cluster backups
│       │   └── minio/        ← MinIO object store data (etcd snapshots + Velero)
│       ├── docker/           ← Docker host appdata backups
│       └── varys/            ← varys key material backups (age key, SSH keys, kubeconfig)
└── tera/
    ├── media/                ← Movies and TV series
    └── downloads/            ← In-progress downloads (active)
```

# Step 1 — Create the Kubernetes Volumes Dataset

This dataset backs all Kubernetes PersistentVolumeClaims. The NFS provisioner will create
one subdirectory per PVC automatically.

**In the TrueNAS web UI:**

1. Navigate to **Storage → Create Dataset**
2. Parent: `core`
3. Name: `k8s_volumes`
4. Share Type: **Generic**
5. Click **Save**

**Or via SSH on TrueNAS:**

```bash
zfs create core/k8s_volumes
```

Verify:

```bash
zfs list core/k8s_volumes
# Should show: core/k8s_volumes with mount point /mnt/core/k8s_volumes
```

---

# Step 2 — Create the Backup Datasets

`archive/backups` is the NFS-exported parent. All backup destinations are nested within it.

**Via SSH on TrueNAS:**

```bash
# Parent backup dataset (NFS exported)
zfs create archive/backups

# Kubernetes cluster backups
zfs create archive/backups/k8s
zfs create -o compression=off -o sync=disabled -o recordsize=1M archive/backups/k8s/minio

# Host backups
zfs create archive/backups/docker
zfs create archive/backups/varys
```

> MinIO sync is disabled deliberately — MinIO has its own internal consistency mechanisms.
> etcd snapshots are written directly to MinIO over S3 (no NFS mount required on k3s nodes).

Verify:

```bash
zfs list | grep backups
# Expected:
# archive/backups                         ...  /mnt/archive/backups
# archive/backups/k8s             ...  /mnt/archive/backups/k8s
# archive/backups/k8s/minio       ...  /mnt/archive/backups/k8s/minio
# archive/backups/docker          ...  /mnt/archive/backups/docker
# archive/backups/varys           ...  /mnt/archive/backups/varys
```

# Step 3 — Configure NFS Exports

Two NFS shares are required. The Kubernetes volumes share is exported to the entire cluster subnet. The backup share is exported to specific hosts that need backup access.

**In the TrueNAS web UI, navigate to Sharing → NFS → Add.**

### Share 1 — Kubernetes Volumes

| Setting | Value |
|---|---|
| Path | `/mnt/core/k8s_volumes` |
| Maproot User | `root` |
| Maproot Group | `wheel` |
| Allowed Networks | `10.0.10.0/24` |
| Enable NFSv4 | Yes |

This allows all nodes on the homelab subnet to mount the volume share. The NFS provisioner
running inside the cluster uses this share to create per-PVC subdirectories dynamically.

### Share 2 — Backup Storage

| Setting | Value |
|---|---|
| Path | `/mnt/archive/backups` |
| Maproot User | `root` |
| Maproot Group | `wheel` |
| Allowed Hosts | `10.0.10.11` (tywin), `10.0.10.13` (jaime), `10.0.10.12` (tyrion), `10.0.10.20` (docker host), `10.0.10.10` (varys) |
| Enable NFSv4 | Yes |

The backup share is locked to specific hosts rather than the full subnet. Only the cluster nodes,
Docker host, and automation host need backup mount access.

After saving each share, restart the NFS service:

**System → Services → NFS → Restart**

### Verify NFS Exports are Active

From the automation host (`varys`, 10.0.10.10):

```bash
showmount -e 10.0.10.80
```

Expected output:

```
Export list for 10.0.10.80:
/mnt/core/k8s_volumes    10.0.10.0/24
/mnt/archive/backups     10.0.10.10 10.0.10.11 10.0.10.13 10.0.10.12 10.0.10.20
```

If `showmount` is not installed:

```bash
sudo apt install -y nfs-common
```

# Step 4 — Deploy MinIO on TrueNAS

MinIO provides an S3-compatible API on top of TrueNAS storage. Velero uses it as the backup
target during Flux bootstrap (`platform-backup` kustomization).

## Install the MinIO App

Navigate to: **Apps → Discover Apps → MinIO → Install**

| Setting | Value |
|---|---|
| Application Name | `minio` |
| Root User | `admin` |
| Root Password | *(generate strong password — store in password manager)* |
| Storage type | Host Path |
| Host Path | `/mnt/archive/backups/k8s/minio` |
| API Port | `9000` |
| Console Port | `9001` |
| Host Network | Enabled |

## Create the Velero Bucket

Once MinIO is running, open the console at `http://10.0.10.80:9001` and log in with the root credentials.

1. Navigate to **Buckets → Create Bucket**
2. Bucket Name: `velero`
3. Versioning: Off
4. Object Locking: Off
5. Click **Create Bucket**

## Create an Access Key for Velero

Velero uses a dedicated access key — never the root credentials.

The TrueNAS MinIO App (Community Edition) exposes an Object Browser console only — there is no Access Keys section in the UI. Use the `mc` (MinIO Client) CLI instead.

**Install `mc` on varys** (amd64 — Intel NUC):

```bash
wget https://dl.min.io/client/mc/release/linux-amd64/mc
chmod +x mc
sudo mv mc /usr/local/bin/mc
```

**Connect `mc` to your MinIO instance:**

```bash
mc alias set truenas http://10.0.10.80:9000 admin YOUR_ROOT_PASSWORD
```

**Verify it can see the bucket:**

```bash
mc ls truenas
# Expected: [date] velero/
```

**Create a dedicated service account for Velero:**

```bash
mc admin user svcacct add truenas admin --name velero-svc
```

Output:

```
Access Key: XXXXXXXXXXXXXXXXXXXX
Secret Key: XXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX
```

**Copy both values now — the secret key is only shown once.** Store both in your password manager. These are the credentials you will use in Guide 03 (Secrets Management) when encrypting the Velero MinIO secret.

## Verify MinIO

```bash
curl -I http://10.0.10.80:9000/minio/health/live
# Expected: HTTP/1.1 200 OK
```

> For the full MinIO reference, see [truenas/docs/minio-configuration.md](../../truenas/docs/minio-configuration.md).

# Step 5 — Install NFS Client Utilities on k3s Nodes

The NFS subdir external provisioner (deployed by Flux in `platform-storage`) requires the
`nfs-common` package on every k3s node. Without it, pods will fail with `mount: bad option;
need helper program` errors when attempting to mount NFS volumes.

> **Note:** This step is also handled automatically by the Ansible playbook
> `ansible/playbooks/lifecycle/install-cluster.yml`, which installs `nfs-common` as part of
> node preparation. If you are following the guides sequentially, the playbook in Guide 02
> will install this automatically. This step is here as a verification checkpoint.

## Verify on each node

```bash
# From varys, check all nodes
ansible -i ansible/inventory/homelab.yml k3s_primary,k3s_servers \
  -m shell -a "dpkg -l nfs-common | grep '^ii'" --become
```

If any node is missing `nfs-common`:

```bash
ansible -i ansible/inventory/homelab.yml k3s_primary,k3s_servers \
  -m apt -a "name=nfs-common state=present update_cache=yes" --become
```

## Verify NFS mount works from a node

SSH into any k3s node and test:

```bash
sudo mkdir -p /tmp/test-nfs
sudo mount -t nfs 10.0.10.80:/mnt/core/k8s_volumes /tmp/test-nfs
ls /tmp/test-nfs
sudo umount /tmp/test-nfs
```

If the mount fails, check that the NFS export allows the node's IP (Step 3).

# Step 6 — Install the GitHub Actions Self-Hosted Runner on varys

The CI/CD pipeline requires a self-hosted runner on `varys` (10.0.10.10). Without LAN access,
the runner cannot perform `flux diff` against the production cluster API or run
`kubeconform` validation steps. GitHub-hosted runners have no route to your private network.

## Create the Runner in GitHub

1. Navigate to your repository on GitHub
2. Go to **Settings → Actions → Runners → New self-hosted runner**
3. Select **Linux** and **x64** (varys is an Intel NUC — x86_64 / AMD64)
4. Follow the displayed commands to download, configure, and start the runner

The commands GitHub shows will look similar to this (use the exact tokens shown in the UI):

```bash
# SSH into varys first
ssh kagiso@10.0.10.10

# Create a directory for the runner
mkdir -p ~/actions-runner && cd ~/actions-runner

# Download the runner package
curl -o actions-runner-linux-x64.tar.gz -L \
  https://github.com/actions/runner/releases/download/v2.333.1/actions-runner-linux-x64-2.333.1.tar.gz

# Extract
tar xzf ./actions-runner-linux-x64.tar.gz

# Configure (use the URL and token GitHub shows you — do not use placeholders here)
./config.sh --url https://github.com/YOUR_ORG/homelab-infrastructure --token YOUR_TOKEN

# Install as a systemd service so it survives reboots
sudo ./svc.sh install
sudo ./svc.sh start
```

## Install Required Tools on varys

The runner must have several tools installed for CI jobs to work. Install them all now.

**kubectl** (amd64):

```bash
curl -LO "https://dl.k8s.io/release/$(curl -L -s https://dl.k8s.io/release/stable.txt)/bin/linux/amd64/kubectl"
sudo install -o root -g root -m 0755 kubectl /usr/local/bin/kubectl
rm kubectl
kubectl version --client
```

**flux CLI** (amd64):

```bash
curl -s https://fluxcd.io/install.sh | sudo bash
flux version
```

**kubeconform** (schema validation tool):

```bash
KUBECONFORM_VERSION=0.6.4
curl -sL "https://github.com/yannh/kubeconform/releases/download/v${KUBECONFORM_VERSION}/kubeconform-linux-amd64.tar.gz" \
  | tar xz
sudo install -m 755 kubeconform /usr/local/bin/kubeconform
rm kubeconform
kubeconform -v
```

**kustomize**:

```bash
curl -s "https://raw.githubusercontent.com/kubernetes-sigs/kustomize/master/hack/install_kustomize.sh" | bash
sudo mv kustomize /usr/local/bin/kustomize
kustomize version
```

**pluto** (deprecated API detector):

```bash
PLUTO_VERSION=5.19.0
curl -sL "https://github.com/FairwindsOps/pluto/releases/download/v${PLUTO_VERSION}/pluto_${PLUTO_VERSION}_linux_amd64.tar.gz" \
  | tar xz
sudo install -m 755 pluto /usr/local/bin/pluto
rm pluto
pluto version
```

**gh CLI** (GitHub CLI — used in CI scripts):

```bash
curl -fsSL https://cli.github.com/packages/githubcli-archive-keyring.gpg \
  | sudo dd of=/usr/share/keyrings/githubcli-archive-keyring.gpg
echo "deb [arch=amd64 signed-by=/usr/share/keyrings/githubcli-archive-keyring.gpg] \
  https://cli.github.com/packages stable main" \
  | sudo tee /etc/apt/sources.list.d/github-cli.list
sudo apt update && sudo apt install -y gh
gh version
```

## Verify Runner is Online

After installation:

```bash
cd ~/actions-runner && sudo ./svc.sh status
# Expected: active (running)
```

In GitHub: **Settings → Actions → Runners** — the runner should show as **Idle** (green dot).

# Step 7 — Configure Branch Protection on main

Branch protection ensures that no commit reaches `main` unless all CI checks pass. This is the
enforcement mechanism for the PR-based validation model — without it, a broken manifest could
be merged directly and cause Flux reconciliation failures.

**Part A — Enable auto-merge at the repository level** (Settings → General):

1. Navigate to your repository → **Settings → General**
2. Scroll down to the **Pull Requests** section
3. Check **Allow auto-merge**
4. Click **Save**

This is a repository-level toggle that makes auto-merge available on individual PRs. It is separate from branch protection.

**Part B — Configure branch protection** (Settings → Branches):

1. Navigate to your repository → **Settings → Branches**
2. Click **Add branch protection rule** (or edit if a rule already exists)
3. Branch name pattern: `main`
4. Enable: **Require status checks to pass before merging**
5. Add required checks: search for and add `Validate manifests`
   > `dry-run` will not appear in the search until it has run at least once. Add it after your first PR.
6. Enable: **Require branches to be up to date before merging**
7. Leave **Require a pull request before merging** unchecked (single-operator repo — no reviewer needed)
8. Click **Save changes**

With this in place, the merge button on any PR is blocked until all required CI checks show green.

# Step 8 — Create the Cloudflare API Token

cert-manager uses a Cloudflare API token to complete the DNS-01 ACME challenge when
issuing the `*.kagiso.me` wildcard TLS certificate. The token is stored in Ansible Vault
(added in Guide 04) and then applied to the cluster as a `kubectl` Secret immediately
after Flux bootstrap — before cert-manager attempts to issue any certificates.

**Required permissions:**

| Zone | Permission |
|---|---|
| Zone → Zone | Read |
| Zone → DNS | Edit |

**Scoped to:** `kagiso.me` only (not all zones).

**Steps:**

1. Log into the Cloudflare dashboard
2. Navigate to **My Profile → API Tokens → Create Token**
3. Start from the **Edit zone DNS** template
4. Set **Zone Resources** to: `Include → Specific zone → kagiso.me`
5. Click **Continue to summary → Create Token**
6. **Copy the token now** — it is only shown once. Store it in your password manager. It will be added to Ansible Vault in [Guide 04 — Saving the Deploy Key to Vault](./04-Flux-GitOps.md#saving-the-deploy-key-to-vault), then used to create a Kubernetes Secret in the `cert-manager` namespace immediately after Flux bootstrap. It is never committed to Git directly.
